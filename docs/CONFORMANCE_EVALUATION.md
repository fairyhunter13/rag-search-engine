## 1. World Model (OSE — governance/spec regime)

### 1.1 June-2026 definition (cited)

OSE uses the **governance/spec regime**: state = codebase + invariants; action = a diff; action-conditioned next-state = conformance verdict; planner = permitted change-sequences; validator = `check_world_model.py`. Full definition: `docs/reference/world-model.md`.

Four-property conformance bar: (1) state representation, (2) action-conditioned next-state (guard + delta), (3) multi-step rollout, (4) planning/counterfactual + validator.

### 1.2 Per-property scorecard

| Property | Status | Evidence |
|---|---|---|
| (1) State representation | ✅ Pass | `model.yaml` L1 (P0–P11 invariants) + L2 (component map) enumerate the "domain" of OSE |
| (2) Action-conditioned next-state | ⚠️ Partial | L1 `check` predicates encode guards; however no explicit `delta:` (effect) field per action — `check_world_model.py` emits CONFORMS/AT_RISK but does not record *what changes* |
| (3) Multi-step rollout | ⚠️ Partial | `check_world_model.py --base HEAD~1 --head HEAD` checks one diff; multi-step sequence checking across N commits not yet implemented |
| (4) Planning / counterfactual | ✗ **Gap** | No `plan(state, goal)` for governance-level "which change-sequences are permitted?"; AT_RISK verdict is reactive not predictive |
| Validator | ✅ Pass | `check_world_model.py` rejects diffs that violate L1 invariants — equivalent governance validator |
| Regime declared | ✅ Pass | `docs/world-model/README.md` states "governance artifact; no MCP surface, no FEATURES.md entry" |
| Skills generated + enforced | ✅ Pass | `.claude/skills/world-model.md` + `.claude/skills/info-hierarchy.md` generated by `gen_world_model_skills.py`; wired in `CLAUDE.md` |
| One shared schema, two profiles | ✗ **Gap** | `check_world_model.py` is OSE-only; does not yet accept `--repo <companion-repo>` for the governance+execution profile |

### 1.3 Gap → remediation map

| Gap | Remediation | Plan phase |
|---|---|---|
| No `delta:` per action | Add `delta:` field to each L1 invariant: what the check *prevents* (the undesired state-transition) | Phase 3 |
| No multi-step sequence check | Extend `check_world_model.py` to accept a list of diffs/commits and evaluate each in sequence | Phase 3 |
| No governance planner | Add `--plan` mode: "given current violations, which fixes are permitted and in what order?" | Phase 3 |
| Checker is OSE-only | Parameterize with `--repo` + `--profile` (governance-only vs governance+execution) | Phase 3 |

## 2. Information Hierarchy (OSE — `docs/info-hierarchy.md`)

### 2.1 June-2026 definition (cited)

IH = value/importance-ranked tree (Wurman LATCH "H"); drill-down/roll-up traversal; distinct from IA (structural framework) and visual hierarchy (rendering). Full definition: `docs/reference/information-hierarchy.md`.

### 2.2 Per-property scorecard

| Property | Status | Evidence |
|---|---|---|
| Value/generality spine is the centerpiece | ⚠️ Partial | `docs/info-hierarchy.md` is a **DIKW doctrine ladder**, not an IH of OSE's codebase domains. It correctly maps OSE's LLM spend per DIKW rung, but this is *not* a per-domain IH (no generality tree with PRIMARY/SECONDARY/TERTIARY levels). |
| Drill-down / roll-up named as §2 | ✗ **Gap** | Not present in `docs/info-hierarchy.md` (DIKW ladder has no drill-down/roll-up) |
| Per-domain leaf files (5-section order) | ✗ **Gap** | OSE has no `docs/information-hierarchy/` with per-domain spine files |
| Visual ranking as §3 | ✗ **Gap** | Not present |
| IA systems subordinate under §4 | ✗ **Gap** | Not present |
| Code-grounded claims (`[code: file:line]`) | ✗ **Gap** | DIKW ladder has no citations |
| `docs/info-hierarchy.md` purpose | ✅ Correct for its purpose | This file serves the **DIKW spend doctrine** role and should be preserved. The *codebase IH* is a separate artifact to be generated by `docgen` (Phase 2) |

### 2.3 Gap → remediation map

| Gap | Remediation | Plan phase |
|---|---|---|
| No per-domain IH files | `docgen` Phase 2 generates `docs/information-hierarchy/` with LLM-native IH (5-section order, code-grounded) | Phase 2 |
| `info-hierarchy.md` is DIKW doctrine, not IH | Rename/clarify: `docs/info-hierarchy.md` → `docs/dikw-doctrine.md` to avoid conflation; update all cross-refs | Phase 2 |
